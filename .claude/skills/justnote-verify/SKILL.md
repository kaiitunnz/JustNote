# Verifying JustNote

JustNote is a macOS accessory (menu-bar-less) app: a global shortcut summons a floating
`NSPanel` hosting the SwiftUI `MenuView`; Settings is a separate AppKit window. The XCTest
suite is **model-only** — every panel/window/keyboard behavior must be verified by *driving the
running app*, because there is no UI test target.

## Build & test (static gate)

The `.xcodeproj` is generated by XcodeGen and git-ignored; `project.yml` is the source of truth.
Regenerate after adding/removing source files (sources are globbed by directory, so no `project.yml`
edit is needed for new files):

```sh
xcodegen generate
xcodebuild -project JustNote.xcodeproj -scheme JustNote -configuration Debug -derivedDataPath build build
xcodebuild test -project JustNote.xcodeproj -scheme JustNote -configuration Debug -derivedDataPath build -destination 'platform=macOS'
```

Expect `** BUILD SUCCEEDED **` and 15 passing tests (all `JustNoteTests`, model-only). The
`com.apple.linkd.autoShortcut` connection warnings in test output are harmless.

## Launch & summon the panel

```sh
open -n build/Build/Products/Debug/JustNote.app        # -n: force a fresh instance
```

A **cold launch opens the Settings window** (no panel). Close it before driving the panel so it
doesn't confuse window-count checks:

```sh
osascript -e 'tell application "System Events" to tell process "JustNote" to click button 1 of window "JustNote Settings"'
```

Summon / toggle the panel with the real global shortcut ⌥. (`key code 47` = period, option down).
`KeyboardShortcuts` uses Carbon `RegisterEventHotKey`, so this works with no Accessibility prompt:

```sh
osascript -e 'tell application "System Events" to key code 47 using {option down}'
```

## Read window state (the observation API)

Use the accessibility API via System Events — coordinates are top-left origin, screen points.

```sh
# window count — THE oracle for summon/dismiss (panel up = 1, dismissed = 0)
osascript -e 'tell application "System Events" to tell process "JustNote" to get count of windows'
# panel frame: {x, y, w, h}
osascript -e 'tell application "System Events" to tell process "JustNote" to get {position, size} of window 1'
# find a header button by its help text and get its frame (buttons live in group 1)
osascript -e 'tell application "System Events" to tell process "JustNote" to get {position, size} of (first button of group 1 of window 1 whose help is "New note")'
```

Button `help` values: `New note`, `Pin note`, `Delete note`, `Hide sidebar`, `Preview markdown`,
`Reveal storage in Finder`, `Settings`.

**AXPress vs. real click:** `click (button … )` via AX invokes the action directly and bypasses
hit-testing — good for "is the button wired", useless for "does a real click reach it under the
titlebar". To test real hit-testing, inject a mouse click at the button's screen center with the
CGEvent tool below.

Note data lives at `~/Library/Application Support/JustNote/` — `Notes/<uuid>.txt` and `state.json`
(`noteOrderIDs`, `notes`, `selectedNoteID`). Reading note count / order there is the way to confirm
create/delete/reorder actually happened.

## Injecting mouse drags & clicks (CGEvent tool)

There is no `cliclick` or PyObjC here, so compile this tiny Swift tool once. It does press → stepped
drag → release (equal start/end = a click):

```swift
// dragtool.swift — usage: dragtool x0 y0 x1 y1   (screen coords, top-left origin)
import CoreGraphics
import Foundation
func post(_ e: CGEvent?) { e?.post(tap: .cghidEventTap) }
func mouse(_ t: CGEventType, _ x: Double, _ y: Double) {
    post(CGEvent(mouseEventSource: nil, mouseType: t, mouseCursorPosition: CGPoint(x: x, y: y), mouseButton: .left))
}
func drag(_ x0: Double,_ y0: Double,_ x1: Double,_ y1: Double) {
    mouse(.mouseMoved, x0, y0); usleep(60000)
    mouse(.leftMouseDown, x0, y0); usleep(120000)
    for i in 1...30 { let t = Double(i)/30; mouse(.leftMouseDragged, x0+(x1-x0)*t, y0+(y1-y0)*t); usleep(12000) }
    usleep(60000); mouse(.leftMouseUp, x1, y1); usleep(120000)
}
let a = CommandLine.arguments
drag(Double(a[1])!, Double(a[2])!, Double(a[3])!, Double(a[4])!)
```

```sh
swiftc dragtool.swift -o dragtool     # write both to the scratchpad, not the repo
./dragtool 1183 684 1283 784          # drag bottom-right corner outward = resize
./dragtool 855 226 855 226            # click at a point
```

Drive the panel by computing coordinates from its frame (parse `{position, size}` above):
title-bar strip (top ~28px, empty areas) drags the window; the bottom-right corner resizes; a
header button's center clicks it.

## Panel regression checklist

Drive these after any change to `PanelController`, `AppDelegate`, or the panel styling:

1. `⌥.` summons a single panel (`count of windows` → 1); `⌥.` again and Escape each dismiss it (→ 0).
2. **First-keystroke** Space and Return after summoning do **not** dismiss the panel (the core bug
   the panel design fixes) and are not absorbed.
3. Delete-confirm (`⌘⇧D`) → Cancel leaves the panel open (count stays 1).
4. **Move:** dragging the title-bar strip moves the window; re-summon keeps the position.
5. **Resize:** dragging the bottom-right corner resizes (clamped to the 480×360 minimum); re-summon
   keeps the size; quit + relaunch restores both (frame autosave).
6. **Note DnD not hijacked:** dragging a sidebar note row does **not** move the window.
7. Cold launch / Dock reopen opens Settings, not the panel.

## Caveats (paid for empirically)

- **`screencapture` is blocked here** (no Screen Recording permission) — you cannot verify *visual*
  chrome (rounded corners, material, the titlebar-inset gap) from the CLI. Those need a human on a
  real display. Verify geometry/behavior via the AX frame reads instead.
- **Driving the app mutates the real user store.** Space/Return can activate the empty-state "Create
  note" button and spawn 0-byte notes in `~/Library/Application Support/JustNote/Notes`. Do **not**
  bulk-delete from that store to "clean up" — it's the user's live data; leave artifacts and tell the
  user (they delete in-app with ⌘⇧D), or ask before removing.
- **SwiftUI drag-and-drop can't be driven** by synthetic CGEvent drags (it needs a real
  `NSItemProvider` pasteboard session), so note *reorder* is a manual-only check; automation can only
  confirm the window isn't hijacking the drag.
- Kill a test instance with `kill "$(pgrep -x JustNote)"` between runs so a stale one doesn't answer
  your AX queries.
